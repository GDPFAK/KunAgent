### Step 3: 专家分析判断

#### 3.1 微服务 Critical Path 分析

```
分析逻辑（40+ 微服务不要逐个看）：

① 生成服务拓扑 → 找 Critical Path（最长调用链）
② 在 Critical Path 上找 P99 > 500ms 的服务
③ 按延迟贡献排序 → 优化 Top 3 影响最大的
④ 调用深度 > 7 层 → 考虑合并微服务

诊断矩阵：
┌──────────────────┬────────────────────────────────┐
│ 现象              │ 可能根因                        │
├──────────────────┼────────────────────────────────┤
│ 所有接口同时变慢  │ 基础设施层（DB/Redis/网络）      │
│ 特定接口慢        │ 该服务自身问题（慢查询/GC/锁）  │
│ 间歇性毛刺        │ GC/连接池耗尽/定时任务冲突       │
│ 链路放大          │ N+1 查询/循环 RPC/扇出过大      │
│ 超时雪崩          │ 未配置熔断，一个慢服务拖垮整条链 │
└──────────────────┴────────────────────────────────┘

关键阈值：
- P99 > 500ms = 需要优化
- P99 > 2s = 严重（用户可感知）
- 错误率 > 1% = 告警
- 错误率 > 5% = 严重
```

#### 3.2 K8s 资源效率

```
从 CloudQ 拿到集群资源数据后：

资源效率判断标准：
┌────────────────────┬───────────────────────────┐
│ 指标                │ 判断                       │
├────────────────────┼───────────────────────────┤
│ request:limit 配比 │ 理想 1:2；< 1:1.5 太保守   │
│                    │ > 1:4 太激进（OOM风险）     │
├────────────────────┼───────────────────────────┤
│ 节点 CPU 利用率    │ 40-70% = 合理              │
│                    │ < 30% = 过度配置（浪费）    │
│                    │ > 80% = 容量风险            │
├────────────────────┼───────────────────────────┤
│ 节点内存利用率     │ 50-80% = 合理              │
│                    │ < 40% = 过度配置            │
│                    │ > 85% = Eviction 风险      │
├────────────────────┼───────────────────────────┤
│ 未使用 PVC         │ Bound 但无 Pod 挂载 = 闲置  │
│ 闲置 namespace     │ 无 Running Pod = 清理       │
└────────────────────┴───────────────────────────┘

优化建议优先级：
1. request 过高的 Pod → 降低 request（立即省钱，零风险）
2. 闲置 PVC/namespace → 清理
3. 低利用率节点 → 开启 Cluster Autoscaler 缩容
4. 无 HPA 的服务 → 配置 HPA
```

#### 3.3 API 网关 P99 三段归因

```
网关 P99 波动时的分析逻辑：

① 看限流触发率
   - 触发率 > 1% = 限流阈值太低（正常请求被拦）
   - 触发率 = 0 → 不是限流问题

② 看上游服务延迟
   - 网关自身延迟（< 5ms 正常）vs 上游延迟
   - 上游延迟增加 → 具体查哪个上游服务
   - 80% 的网关 P99 波动来自上游服务传导

③ 看配额使用率
   - QPS 配额 > 80% = 接近上限
   - 连接数配额 > 80% = 需扩规格

经验值：
- 网关自身延迟应 < 5ms（如果 > 20ms → 网关自身问题）
- 限流应该是最后防线，不应该日常触发
```

#### 3.4 容器化迁移 ROI

```
评估模型：

迁移优先级评分 = 无状态(+3) + 高流量(+2) + 依赖简单(+2) + 频繁发布(+1)

分类：
- 得分 ≥ 6 = 第一批迁移
- 得分 4-5 = 第二批
- 得分 < 4 = 暂不迁移
- 有状态服务（DB/Redis/MQ）= 最后迁或不迁

ROI 计算：
年化收益 = (虚机月费 - 容器月费) × 12 + 运维人力节省
迁移成本 = 改造工时 × 人天成本 + 迁移期间冗余资源费
回本周期 = 迁移成本 ÷ 月节省

判断：回本 < 6 个月 = 值得迁；6-12 个月 = 可迁；> 12 个月 = 暂缓
```

## 注意事项

- 40+ 微服务不要逐个看，从 Critical Path 切入
- 相对激进的方案（合并微服务、降低冗余）会标注风险
- HPA 不生效先查 metrics-server 是否部署
- 容器迁移优先无状态服务，有状态最后迁
- 节点利用率看 7 天趋势，不看瞬时值
