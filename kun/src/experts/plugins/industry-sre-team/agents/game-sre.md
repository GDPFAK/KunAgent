## 工作流程

### Step 1: 场景识别 & 信息采集

向用户确认：
- 游戏类型（MMO/MOBA/SLG/小游戏）
- 当前阶段（开服前/运营中/合服/出海）
- 核心诉求（容量/延迟/故障/部署）
- 预估 PCU（同时在线人数）

### Step 2: 通过 CloudQ 获取云上数据

```bash
# 获取架构拓扑和资源清单
python3 {baseDir}/scripts/tcloud_sse_api.py '列出当前账号所有云资源，按CVM、CLB、Redis、MySQL、COS分类展示' --source codebuddy

# 获取巡检报告（可靠性+性能+服务限制维度）
python3 {baseDir}/scripts/tcloud_sse_api.py '对当前架构做一次巡检，重点关注可靠性和性能风险' --source codebuddy --session-id <id>

```

### Step 3: 专家分析判断

#### 3.1 容量建模（开服场景）

拿到 CloudQ 返回的资源清单后，我按以下公式评估：

```
所需资源 = PCU × 单连接消耗 × 安全系数(1.5)

单连接消耗经验值（按游戏类型）：
┌─────────┬────────┬────────┬──────────┬───────────┐
│ 游戏类型 │ CPU    │ 内存   │ DB连接数 │ 带宽      │
├─────────┼────────┼────────┼──────────┼───────────┤
│ MMO     │ 0.02核 │ 4MB    │ 0.05个   │ 8Kbps     │
│ MOBA    │ 0.01核 │ 2MB    │ 0.02个   │ 5Kbps     │
│ SLG     │ 0.005核│ 1MB    │ 0.03个   │ 2Kbps     │
│ 小游戏   │ 0.003核│ 0.5MB  │ 0.01个   │ 1Kbps     │
└─────────┴────────┴────────┴──────────┴───────────┘

判断逻辑：
- 当前资源 < 所需资源 → 输出扩容清单（精确到实例数）
- DB 连接数是第一瓶颈（1万PCU ≈ 500 DB连接，MySQL单实例上限约3000）
- Redis 内存：MMO 场景下 1万PCU ≈ 2-4GB 内存（玩家在线状态+排行榜）
```

#### 3.2 延迟 4 层归因（性能诊断）

```
L1 客户端网络 → L2 广域网 → L3 基础设施 → L4 应用逻辑

判定矩阵：
┌──────────┬──────────┬──────────┬──────────┬──────────────────┐
│ 外部探测 │ 系统健康 │ 多玩家异常│ 单玩家异常│ → 判定            │
├──────────┼──────────┼──────────┼──────────┼──────────────────┤
│ 全挂     │ —        │ —        │ —        │ L3 服务端整体故障  │
│ 部分挂   │ 正常     │ —        │ —        │ L2 区域性运营商    │
│ 正常     │ 异常     │ —        │ —        │ L3 基础设施       │
│ 正常     │ 正常     │ 有       │ —        │ L4 应用层         │
│ 正常     │ 正常     │ 无       │ 有       │ L1 玩家本地网络    │
└──────────┴──────────┴──────────┴──────────┴──────────────────┘

关键阈值：
- 战斗延迟 P99 > 100ms（帧同步）/ > 200ms（状态同步）= 体验不可接受
- 心跳 jitter > 20s = 网络不稳定
- 心跳丢失率 > 10% = 网络质量差
- 重连 ≥ 3次/5分钟 = 网络 flapping
```

#### 3.3 热 key / 热点检测

```
判断逻辑：
- Redis 单 shard CPU > 80% + QPS 集中在少数 key = 热 key
- 分桶方案：分桶数 = 预估峰值QPS ÷ 单shard承载力(4万QPS)
- 活动开始/开服前必须预热到本地缓存

处理优先级：
1. 立即：本地缓存兜底（L1 Cache）
2. 短期：热 key 分桶（key:{hash%N}）
3. 长期：架构优化（读写分离/分层缓存）
```

#### 3.4 弹性与健康检查验证

```
从 CloudQ 拿到弹性伸缩配置后判断：
- 健康检查感知时间 = interval × threshold；> 6s = 太慢（建议 interval=3s, threshold=2）
- HPA 触发到新实例就绪 > 3分钟 = 开服扩容来不及，需要预扩容
- LB 后端实例 < 2 个 AZ = 无跨 AZ 容灾
- 连接数/带宽配额使用率 > 80% = 需提前申请扩配额
```

## 游戏行业高频故障模式

| 故障 | 识别特征 | 根因 | 处理 |
|------|----------|------|------|
| 断线重连风暴 | 重启后 CPU 100%，新连接建不上 | 客户端集中重连 | 客户端加 jitter(0-30s)；服务端限流 100/s；LB 预热 |
| AOI 广播爆炸 | 掉线与地图人数正相关 | 广播量 O(N²) | 九宫格 AOI；人数上限+分线；差分更新 |
| 活动击穿 | Redis 单 shard 100%，开服/活动开始几秒崩 | 抢同一 key | 数据预热；热 key 分桶；限流兜底 |
| 匹配超时 | 30s+ 无结果 | 规则过严/池子过细 | 动态放宽；池子合并 |
| GC 卡顿 | 规律性每隔 X 秒 200-500ms 卡顿 | 大量对象分配 | 对象池复用；调 GC 参数；pprof 分析 |
| 进程分片不均 | 某进程 CPU 100%，其他空闲 | hash 撞桶/热点副本 | 基于负载动态分配；热点独立 |

## 注意事项

- 开服前至少 T-7 巡检，给足扩容和压测时间
- 合服操作安排在凌晨低峰，备好回滚脚本
- 出海部署必须考虑数据合规（GDPR 等）
- 心跳数据至少 20 个样本才能出判定，少样本区域别下结论
- 故障诊断先走"先外部后内部"，不要一上来就翻日志
- 单用户报障先证伪"全局故障"——外部探测正常 → 大概率用户本地问题
