### Step 3: 专家分析判断

#### 3.1 全链路卡顿三段归因

```
三段切分模型：
主播推流 → 源站/转码 → CDN分发/播放端

判断逻辑：
┌────────────────┬───────────────────────────────────┐
│ 段落           │ 卡顿特征 → 根因                    │
├────────────────┼───────────────────────────────────┤
│ 推流端(主播)   │ 上行丢帧/码率波动 → 主播网络差     │
│               │ 编码延迟高 → 编码器配置/设备性能    │
├────────────────┼───────────────────────────────────┤
│ 源站/转码      │ 转码延迟 > 2s → 转码资源不足       │
│               │ 源站 5xx 增加 → 源站过载            │
├────────────────┼───────────────────────────────────┤
│ CDN/播放端     │ 特定地域卡 → CDN 节点异常           │
│               │ 所有地域卡 → 回源异常/缓存失效      │
│               │ 只有特定用户 → 播放端网络/设备       │
└────────────────┴───────────────────────────────────┘

关键阈值：
- 卡顿率目标 < 2%（> 5% = 严重影响体验）
- 首帧时长目标 < 1s（> 2s = 用户流失）
- 推流成功率 > 99.5%
- 转码延迟 < 2s

卡顿归因三板斧：
1. 只有特定主播卡 → 推流端问题
2. 所有主播都卡 → 源站/转码问题
3. 特定地域/运营商卡 → CDN 节点问题
```

#### 3.2 CDN 成本优化

```
从 CloudQ 拿到 CDN 数据后分析：

核心指标：
┌──────────────┬──────────────┬──────────────────────┐
│ 指标          │ 目标         │ 未达标原因             │
├──────────────┼──────────────┼──────────────────────┤
│ 缓存命中率   │ > 95%        │ URL参数变化/TTL太短   │
│ 回源比       │ < 10%        │ 节点被摘/源站限流      │
│ 带宽利用率   │ 月95计费最优 │ 峰值过高拉高95线       │
└──────────────┴──────────────┴──────────────────────┘

优化分析逻辑：
① 命中率 < 95%：
   - 检查 URL 是否带时间戳/随机参数 → 去参数缓存
   - 检查 Cache TTL 是否太短 → 延长（直播流 TTL = GOP 时长）
   - 检查 Vary 头是否过多 → 精简

② 回源比 > 10%：
   - 检查是否有节点被健康检查摘除 → 修复节点
   - 检查源站是否限流 → 扩容或加源站
   - 检查是否有新上线的流没有预热 → 提前预热

③ 成本优化公式：
   命中率每提升 1% ≈ 减少回源带宽 10%
   月 CDN 费 500 万时：命中率从 93% → 96% ≈ 月省 150 万

④ 计费方式选择：
   - 流量稳定 → 月95带宽计费
   - 流量波动大 → 日峰值计费
   - 有明确峰值活动 → 预付费带宽包
```

#### 3.3 低延迟方案选型

```
延迟预算拆解：
编码延迟 + 传输延迟 + CDN缓存延迟 + 播放缓冲 = 端到端延迟

方案对比：
┌──────────┬──────────────┬─────────────┬──────────────┐
│ 方案      │ 端到端延迟   │ 适用场景     │ 成本(vs标准)  │
├──────────┼──────────────┼─────────────┼──────────────┤
│ HLS      │ 6-30s        │ 点播/回看    │ 基准          │
│ FLV      │ 3-6s         │ 秀场/电商    │ 1x           │
│ LEB      │ 0.5-1s       │ 互动/连麦    │ 1.5x         │
│ WebRTC   │ < 500ms      │ 实时互动     │ 2-3x         │
└──────────┴──────────────┴─────────────┴──────────────┘

选型逻辑：
- 需要 < 1s → 必须 LEB 或 WebRTC（HLS/FLV 不可能）
- 需要 < 500ms → 只能 WebRTC
- GOP 设置：LEB 要求 GOP = 1-2s（太长影响首帧和延迟）
- 缓冲策略：低延迟场景设 0 缓冲（抖动用丢帧补偿）
```

#### 3.4 跨境直播方案

```
跨境延迟分析：
- 中国→东南亚 公网 RTT：80-150ms
- 中国→欧美 公网 RTT：200-300ms
- 中国→南美 公网 RTT：300-400ms

内网加速效果：公网 RTT 减少 30-50%

方案设计：
① 源站就近：主播在哪里，源站就放哪里
② 跨境加速：源站→海外 CDN 用内网中转（不走公网）
③ 多区域源站：大区各放一个源站 + 就近分发
④ 容灾切换：主源站挂 → 自动切备用源站（DNS 故障切换）

关键判断：
- 跨洋公网 RTT > 200ms + FLV 方案 = 必然卡顿
- 内网加速可将延迟降至 80-120ms（勉强可用）
- 需要流畅体验 → 必须海外就近源站
```

## 注意事项

- 头部主播开播是峰值时刻，不在此时做变更
- CDN 命中率优化先看 URL 参数是否导致缓存失效
- 回源带宽飙升先查是否有节点异常被摘除
- 跨境直播需考虑内容合规（各国审核要求不同）
- LEB 对 GOP 要求严格（1-2s），太长首帧和延迟都会差
