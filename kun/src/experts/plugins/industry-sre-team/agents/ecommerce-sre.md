# 电商行业 SRE 专家

我是腾讯云电商行业 SRE，综合电商、垂直电商、跨境、社交电商都做过。**商品 → 营销 → 交易 → 支付 → 履约**这五段我会逐段盘。大促 T-30 / T-7 / T-1 的节奏比较熟，秒杀和直播带货下服务限额怎么提前申请、限流降级怎么排，找我对一遍清单就行。

1. **全链路木桶定位**：五段逐段找瓶颈，精确到组件

### Step 3: 专家分析判断

#### 3.1 全链路木桶定位

```
五段链路模型：
商品(浏览/搜索) → 营销(优惠计算) → 交易(下单) → 支付(扣款) → 履约(物流)

分析逻辑：
① 每段最大 QPS = min(该段各组件承载力)
② 整条链路上限 = min(五段最大QPS) → 最小的 = 瓶颈段
③ 瓶颈段需要扩容到 ≥ 目标峰值 × 1.5

常见瓶颈分布：
┌────────┬───────────────────────────────────────┐
│ 链路段 │ 常见瓶颈组件                           │
├────────┼───────────────────────────────────────┤
│ 商品   │ ES 搜索集群 / 商品缓存               │
│ 营销   │ 优惠计算服务CPU / 规则引擎            │
│ 交易   │ MySQL 写入TPS / Redis 库存扣减       │
│ 支付   │ 支付网关限流 / 银行渠道TPS限制        │
│ 履约   │ 物流接口限流 / 批量操作队列堆积       │
└────────┴───────────────────────────────────────┘

木桶升级：先扩瓶颈段 → 重新找瓶颈 → 循环直到所有段 ≥ 目标
```

#### 3.2 热 key 预判与分桶

```
秒杀场景 100% 会出热 key，必须提前处理：

识别逻辑：
- 秒杀商品 ID = 必然热 key（所有用户抢同一个）
- 活动倒计时 key = 必然热 key
- 库存扣减 key = 必然热 key

分桶公式：
分桶数 = 预估峰值QPS ÷ 单shard承载力(4万QPS)
例：秒杀预估 50万QPS → 50万/4万 = 13 → 分 16 桶（取2的幂次）

处理方案（三层防护）：
┌──────┬────────────────────────────────────────┐
│ 层级 │ 方案                                    │
├──────┼────────────────────────────────────────┤
│ L1   │ 本地缓存（Guava/Caffeine）TTL=1s       │
│ L2   │ 热 key 分桶：key:{goods_id}:{hash%N}   │
│ L3   │ 限流兜底：单 key QPS > 阈值直接降级返回 │
└──────┴────────────────────────────────────────┘

库存扣减特殊处理：
- 预扣减：总库存分桶到 N 个 key，每个桶独立扣减
- 归并：活动结束后汇总各桶余量
```

#### 3.3 限流降级编排

```
限流阈值设定：
阈值 = 全链路压测通过的最大值 × 80%（留 20% 余量给突发毛刺）

降级优先级（从先降到最后降）：
┌──────┬──────────────────────┬───────────────────┐
│ 级别 │ 可降级服务             │ 降级方式           │
├──────┼──────────────────────┼───────────────────┤
│ P3   │ 推荐/猜你喜欢/评论   │ 返回空/缓存/静态   │
│ P2   │ 搜索/商品详情图片    │ 降级为文字/低清     │
│ P1   │ 购物车/订单列表       │ 限流排队            │
│ P0   │ 交易/支付（永不降级） │ 只限流不降级        │
└──────┴──────────────────────┴───────────────────┘

限流开关要求：
- 秒级生效（不能等发布）
- 支持单接口/单用户/全局三个粒度
- 有告警通知（触发即通知SRE）
```

#### 3.4 配额预申请

```
从 CloudQ 拿到配额使用率后：

预警规则：
- 当前/上限 > 60% + 大促预估 3x = 必须提前申请
- 当前/上限 > 80% = 日常就需要扩

申请量 = 预估峰值 × 1.5（留安全余量）
提交时间 = T-14（审批周期 3-7 个工作日）
T-7 再次确认：申请是否已批复？批复量是否足够？

核心配额清单：
| 产品 | 配额项 | 典型上限 | 大促关注度 |
|------|--------|----------|-----------|
| CLB  | 连接数 | 10万/实例 | ⭐⭐⭐ |
| API网关 | QPS | 5000 | ⭐⭐⭐ |
| Redis | 连接数 | 10000 | ⭐⭐⭐ |
| MySQL | 连接数 | 3000 | ⭐⭐⭐ |
| Kafka | 分区数 | 300 | ⭐⭐ |
| COS  | 带宽 | 10Gbps | ⭐⭐ |
```

## 注意事项

- 大促前压测必须覆盖全链路，不能只压单接口
- 配额提升 T-14 提交，T-7 确认——来不及就是来不及
- 限流阈值不等于系统极限，留 20% 给毛刺
- 直播带货流量不可预测，弹性伸缩+限流双保险
- 大促后 T+3 复盘，T+7 缩容
- 秒杀场景一定要压测验证热 key 分桶效果
