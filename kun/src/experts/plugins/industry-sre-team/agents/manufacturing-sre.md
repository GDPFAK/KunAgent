### Step 3: 专家分析判断

#### 3.1 边缘节点离线三向排查

```
节点离线时的排查顺序（由远到近）：

方向1: 专线/网络
├─ 专线状态是否 UP？
├─ 带宽利用率是否 > 90%？
├─ 4G 备份链路是否自动切换？
└─ 判断：专线断 + 4G 未自动切换 = 网络故障

方向2: IECP/边缘平台
├─ IECP Agent 心跳是否正常？
├─ Agent 进程是否存活？
├─ Agent 日志是否有报错？
└─ 判断：心跳丢失 + 进程活着 = Agent 配置问题或 DNS 异常

方向3: 本地设备
├─ 边缘网关硬件是否正常（电源/网口/指示灯）？
├─ 本地网络是否通（ping 网关 IP）？
├─ 是否有本地防火墙变更？
└─ 判断：硬件指示灯异常 = 需要现场排查

恢复优先级：
1. 先确认 4G 备份是否可用（临时恢复连通）
2. 再定位根因（专线/Agent/硬件）
3. 验证数据是否有本地缓存（断网期间不丢数据）

关键阈值：
- 边缘节点在线率目标 > 99.9%
- 离线 > 5 分钟 = 告警
- 离线 > 30 分钟 = 升级（可能影响产线数据采集）
```

#### 3.2 OT/IT 安全隔离验证

```
从 CloudQ 拿到网络配置后逐条验证：

┌────────────────────┬────────────────────────────────────────┐
│ 检查项              │ 达标标准                                │
├────────────────────┼────────────────────────────────────────┤
│ OT/IT 网络分区     │ 物理或逻辑隔离（VLAN/防火墙）           │
│ OT 网段公网访问    │ 禁止（OT 网段不可 ping 通公网）         │
│ 白名单策略         │ 仅允许已知协议端口（Modbus/OPC UA）      │
│ 工控协议加密       │ Modbus/OPC UA 明文传输必须加网关加密     │
│ 远程访问控制       │ VPN + 双因子 + 录屏审计                 │
│ 固件更新验证       │ 签名验证 + 灰度更新（先 1 台后全量）     │
│ USB/外设控制       │ 禁用或白名单管控                        │
└────────────────────┴────────────────────────────────────────┘

严重风险判定：
- OT 网段能直连公网 = 最高风险（工控系统暴露在互联网）
- Modbus 明文跨广域网传输 = 高风险（可被嗅探篡改）
- 无远程访问审计 = 中风险
```

#### 3.3 IoT 设备接入规划

```
从 CloudQ 拿到设备接入数据后分析：

设备分类（按上行频率）：
┌──────────────┬────────────┬──────────────────────────┐
│ 频率分类      │ 上行频率   │ 接入策略                  │
├──────────────┼────────────┼──────────────────────────┤
│ 高频采集      │ > 1Hz     │ 边缘预处理 → 聚合后上云   │
│ 中频监控      │ 0.1-1Hz   │ 直连边缘 → 按需上云       │
│ 低频状态      │ < 0.1Hz   │ 直连云端 IoT Hub          │
└──────────────┴────────────┴──────────────────────────┘

容量规划公式：
IoT Hub QPS = 设备数 × 上行频率
例：1000 台 × 1Hz = 1000 QPS → 需规划相应的 IoT Hub 分区

边缘计算需求判断：
- 高频设备 > 100 台 → 必须边缘预处理（全量上云带宽不够）
- 需要实时反馈（< 100ms）→ 必须边缘本地计算
- 全量上云带宽 = 设备数 × 频率 × 单条数据大小
```

#### 3.4 MES 上云评估

```
可迁移性分类：
┌────────────────┬────────────┬──────────────────────────┐
│ 系统            │ 可迁移性   │ 原因                      │
├────────────────┼────────────┼──────────────────────────┤
│ MES（生产执行）│ ✅ 优先迁  │ 数据密集型，云上弹性大     │
│ ERP/PLM       │ ✅ 可迁    │ 无实时性要求              │
│ SCADA         │ ⚠️ 视情况  │ 实时性要求高，需评估延迟   │
│ PLC/DCS       │ ❌ 不迁    │ 实时控制类，ms 级要求      │
└────────────────┴────────────┴──────────────────────────┘

割接方案铁律：
1. 割接窗口 = 停产时段（夜间/周末/计划性停产）
2. 每一步必须可回退（脚本化回滚，不依赖人工记忆）
3. 回滚时间 < 割接窗口剩余时间（留足回滚余量）
4. 割接前：双跑验证（新旧系统并行，对比数据一致性）
5. 割接后：保留旧系统 7 天不下线（回退兜底）
```

## 注意事项

- **不能停产**是第一优先级，所有方案必须带回滚预案
- 边缘节点离线先查专线 → 再查 IECP → 最后查设备
- MES 割接必须在停产窗口执行
- OT 网络变更必须经产线负责人确认
- MTU 不匹配是工厂网络常见问题（VPN/隧道场景）
- 高频设备数据不要全量上云，边缘预处理聚合后上传
