### Step 3: 专家分析判断

#### 3.1 多租户隔离验证

```
三层隔离逐层验证：

数据层隔离：
┌──────────────────┬──────────────┬───────────────────────────┐
│ 隔离模式          │ 隔离强度     │ 验证方法                   │
├──────────────────┼──────────────┼───────────────────────────┤
│ 行级隔离          │ 弱          │ 所有SQL是否强制带tenant_id？│
│ （同DB同表）      │             │ 有无绕过查询的路径？       │
├──────────────────┼──────────────┼───────────────────────────┤
│ Schema级隔离      │ 中          │ 连接池是否按租户分离？     │
│ （同DB不同Schema）│             │ DDL 是否互相影响？         │
├──────────────────┼──────────────┼───────────────────────────┤
│ DB级隔离          │ 强          │ 每租户独立DB实例？         │
│ （独立DB）        │             │ 网络是否可互访？           │
└──────────────────┴──────────────┴───────────────────────────┘

计算层隔离：
- 租户间 CPU/内存是否互相影响？
- 是否有 cgroup/namespace 隔离？
- 大查询是否会拖慢其他租户？

网络层隔离：
- 租户间是否可以互相访问？
- API 路由是否严格校验 tenant_id？
- 跨租户请求是否有审计？

任意一层失败 = 高危：
- 行级隔离 + SQL 未强制带 tenant_id = 数据泄露风险
- 无计算隔离 + 共享连接池 = Noisy Neighbor 风险
- 网络可互访 = 严重安全隐患
```

#### 3.2 Noisy Neighbor 检测与控制

```
从 CloudQ 拿到资源利用率数据后分析：

检测逻辑：
① 识别资源消耗 Top 5 租户
② 计算占比：Top1 占用 / 总共享资源 × 100%
③ 影响半径验证：大客户负载高时，其他租户 P99 是否恶化？

判断标准：
┌──────────────────┬───────────────────────────────────┐
│ 占比              │ 判定                               │
├──────────────────┼───────────────────────────────────┤
│ 单租户 < 30%     │ 正常                               │
│ 单租户 30-50%    │ 关注（需要监控趋势）               │
│ 单租户 > 50%     │ 危险（必须隔离加固）               │
└──────────────────┴───────────────────────────────────┘

DB 连接数是第一个被抢占的资源：
- 大客户占 60% 连接数 → 其他租户高峰时连接不够用
- 典型表现：其他租户偶发 "too many connections" 但大客户正常

控制方案（按严重度递进）：
1. 限流：每租户 QPS 上限
2. 优先级队列：大客户请求进低优先级队列
3. 独立连接池：大客户独立 DB 连接池
4. 物理隔离：大客户迁移到独立部署
```

#### 3.3 租户级 SLA 计算

```
SLA 计算公式：
SLA = (总时间 - 故障时间 - 计划维护) ÷ (总时间 - 计划维护) × 100%

SLA 等级对照：
┌──────────┬────────────────────┬─────────────────────┐
│ SLA 承诺 │ 月允许停机          │ 年允许停机           │
├──────────┼────────────────────┼─────────────────────┤
│ 99.9%    │ 43.2 分钟          │ 8.76 小时           │
│ 99.95%   │ 21.6 分钟          │ 4.38 小时           │
│ 99.99%   │ 4.32 分钟          │ 52.56 分钟          │
└──────────┴────────────────────┴─────────────────────┘

逐租户统计方法：
① 统计周期：30 天滚动
② 故障时间：该租户请求错误率 > 5% 的分钟数
③ 排除计划维护窗口
④ 大客户需单独统计（不混入全局 SLA）
⑤ 故障归因必须到租户粒度（是全局故障还是只影响特定租户）

未达标时的分析：
- SLA 跌破 → 定位哪次故障导致 → 故障根因是什么
- 根因分类：基础设施(影响所有租户) / Noisy Neighbor(影响邻居) / 租户自身(误操作)
```

#### 3.4 配额管理巡检

```
逐租户配额使用率分析：

预警分级：
┌──────────────┬───────────────────────────────────┐
│ 使用率        │ 动作                               │
├──────────────┼───────────────────────────────────┤
│ < 60%        │ 正常，无需动作                     │
│ 60-80%       │ 黄色预警：记录趋势，预估到红线时间  │
│ 80-95%       │ 橙色预警：通知租户 + 建议升级       │
│ > 95%        │ 红色预警：限流保护 + 紧急扩配       │
└──────────────┴───────────────────────────────────┘

核心配额项：
| 配额项 | 典型限制 | Noisy Neighbor 影响 |
|--------|----------|-------------------|
| DB 连接数 | 按租户分配 | 高（连接耗尽影响所有人）|
| API QPS | 按租户限流 | 高 |
| 存储空间 | 按租户配额 | 中 |
| 文件上传大小 | 全局限制 | 低 |

配额爆了不限流 = 影响其他租户
限流太严 = 影响客户体验
平衡点 = 配额 × 80% 开始限流排队，100% 硬拒绝
```

## 注意事项

- 多租户数据隔离是红线，必须验证跨租户不可访问
- 大客户占用 > 50% 共享资源时应考虑独立部署
- SLA 计算需排除计划内维护窗口
- 配额接近上限（>80%）的租户需提前沟通扩容
- 租户级监控粒度 ≥ 分钟级，否则故障定位不到租户
- 审计日志保留 ≥ 180 天（合规要求）
- DB 连接数是 Noisy Neighbor 的第一受害者
