### Step 3: 专家分析判断

#### 3.1 长连接容量建模

```
容量公式：
所需网关实例数 = 目标同时在线 ÷ 单实例连接上限 × (1 + 30%余量)

经验值：
┌──────────────────────┬─────────────┐
│ 参数                  │ 经验值       │
├──────────────────────┼─────────────┤
│ 单网关实例连接上限    │ 10 万       │
│ 单连接内存消耗        │ 10-50KB     │
│ 心跳间隔              │ 30-60s      │
│ 心跳超时踢线          │ 3次失败     │
│ idle timeout(LB)     │ > 心跳间隔   │
└──────────────────────┴─────────────┘

判断逻辑：
- 当前网关实例数 × 10万 × 70%（留余量）= 安全容量
- 目标在线 > 安全容量 → 需要扩容网关
- LB idle timeout ≤ 客户端心跳间隔 → 连接会被 LB 先断（偶发 RST）

百万在线规划示例：
100万在线 ÷ 10万/实例 = 10 实例（基础）
× 1.3（30%余量）= 13 实例
跨 2 个 AZ 部署：每 AZ 7 实例
```

#### 3.2 消息延迟逐段归因

```
四段模型：
发送 → 中转(路由) → 扩散(群/广播) → 推送(APNs/FCM/厂商通道)

各段延迟目标和判断：
┌──────────┬────────────┬────────────────────────────────┐
│ 段落      │ P99目标    │ 超标时排查                      │
├──────────┼────────────┼────────────────────────────────┤
│ 发送→中转 │ < 50ms    │ 网关→MQ 延迟 / 网络抖动         │
│ 中转→扩散 │ < 100ms   │ 路由查询慢 / 扩散队列堆积       │
│ 扩散→推送 │ < 100ms   │ 推送通道拥塞 / 厂商限流         │
│ 端到端    │ < 300ms   │ 以上各段累加                    │
└──────────┴────────────┴────────────────────────────────┘

群消息扩散模型选择：
┌──────────┬──────────────┬──────────────────────────┐
│ 群规模    │ 推荐模型      │ 原因                      │
├──────────┼──────────────┼──────────────────────────┤
│ < 500人  │ 写扩散        │ 每条消息写 N 份，读取快    │
│ 500-5000 │ 混合         │ 在线用户推送+离线用户拉取  │
│ > 5000   │ 读扩散(时间线)│ 写扩散 O(N) 成本不可接受   │
└──────────┴──────────────┴──────────────────────────┘

万人群写扩散的问题：
- 1条消息 × 10000人 = 10000次写入 → 延迟 O(N)
- 改为读扩散：只写 1 条到时间线 → 用户拉取时读取
```

#### 3.3 瞬时洪峰预案（红包雨/热点事件）

```
洪峰特征：
- 红包雨：1 秒内所有群同时触发，峰值 = 日常 10-50x
- 热点事件：分钟级爬坡到峰值
- 跨年/新年祝福：0:00 瞬时爆发

预案验证清单：
┌────────────────────┬──────────────────────────────────┐
│ 验证项              │ 达标标准                          │
├────────────────────┼──────────────────────────────────┤
│ 弹性扩容速度       │ 10 分钟内扩到 10x                │
│ 预扩容时间         │ 峰值前 30 分钟就位               │
│ 限流分级策略       │ 有 3 级以上分级，可秒级切换       │
│ 降级开关           │ 覆盖所有非核心功能，1s 生效       │
│ MQ 堆积容忍        │ 堆积 100 万条不丢消息             │
└────────────────────┴──────────────────────────────────┘

限流分级策略（从先限到最后限）：
Level 1：限制表情包/已读回执/在线状态同步
Level 2：限制图片/视频消息（只允许文本）
Level 3：限制群消息（只保障单聊）
Level 4：文本消息也限流排队（极端情况）

经验：
- 红包雨峰值 = 群数 × 群内参与率 × 同时发包率
- 提前 30 分钟预扩容 → 避免峰值时扩容来不及
- 断线重连加 jitter（0-30s 随机延迟）→ 避免连接风暴
```

#### 3.4 配额预警分析

```
从 CloudQ 拿到配额数据后判断：

┌──────────────────┬────────────┬────────────────────────┐
│ 配额项            │ 预警阈值   │ 后果                    │
├──────────────────┼────────────┼────────────────────────┤
│ CLB 连接数       │ > 80%     │ 新用户连不上            │
│ Kafka 分区消费堆积│ > 10万    │ 消息延迟飙升            │
│ 网关并发连接     │ > 70%     │ 需要扩容网关实例         │
│ Redis 连接数     │ > 80%     │ 新连接被拒/超时          │
│ 带宽配额         │ > 80%     │ 消息/推送延迟增加        │
└──────────────────┴────────────┴────────────────────────┘

预警 → 行动的逻辑：
- 黄色(>60%)：记录趋势，预估到达红线时间
- 橙色(>80%)：申请扩配额 / 规划扩容
- 红色(>90%)：立即扩容 / 启动限流保护
```

## 注意事项

- 消息"必达"要求极高，可靠性优先于延迟
- 万人群扩散用读扩散/时间线模型，不要写扩散
- 春节红包雨至少 T-30 开始准备
- LB idle timeout 必须 > 客户端心跳间隔
- 推送通道故障先查厂商状态页，再查自有链路
- 断线重连必须加 jitter，避免连接风暴
