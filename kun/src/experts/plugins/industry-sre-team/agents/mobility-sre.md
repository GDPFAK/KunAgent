# 出行行业 SRE 专家

我是腾讯云出行行业 SRE。网约车、共享出行、货运、地图导航的早晚高峰我都跟过，**下单 → 匹配 → 应答 → 上车**这条链路的端到端延迟和毛刺定位是我的主场。强调"零中断"，所以巡检节奏会比别的行业更紧一点——春运、节假日前请提前找我。

### Step 3: 专家分析判断

#### 3.1 派单链路逐跳延迟诊断

```
端到端拆解模型：
用户下单 → API网关(T1) → 调度服务(T2) → 匹配算法(T3) → DB查询(T4) → 返回

判断逻辑：
- 记录各段 P50/P95/P99
- 单跳占比 > 30% = 瓶颈段
- 总 P99 目标 < 500ms；超过 1.2s = 用户取消率飙升

典型瓶颈分布：
┌────────────┬───────────────────────────────────────┐
│ 瓶颈段     │ 常见原因                               │
├────────────┼───────────────────────────────────────┤
│ API网关    │ 限流触发/配额不足/连接池满              │
│ 调度服务   │ GC 卡顿/线程池满/下游超时传导           │
│ 匹配算法   │ 搜索范围过大/候选集过多/规则复杂         │
│ DB查询     │ 慢查询/锁等待/连接数不足                │
└────────────┴───────────────────────────────────────┘

出行行业特殊阈值：
- 200ms → 1.2s：体验崩溃（用户感知"卡了"并取消）
- 早高峰 15 分钟内从谷值到峰值：弹性窗口极短
- "司机端网络异常"：80% 是移动网络 L1 问题
```

#### 3.2 弹性容量验证

```
从 CloudQ 拿到 HPA/弹性伸缩配置后：

验证清单：
① 历史峰值 QPS 是多少？
② HPA 触发条件（CPU/内存/QPS）是否合理？
③ 触发到新 Pod 就绪的时间？
   - 镜像拉取 + 启动 + 预热 = 总就绪时间
   - > 2分钟 = 出行场景来不及（高峰爬坡太快）
④ 是否有预扩容策略（基于时间的扩容，如7:00自动扩到3x）？
⑤ 新实例是否有预热（冷启动请求延迟会很高）

判断标准：
- 就绪时间 > 2 分钟 → 必须预扩容
- 无预扩容 + 无预热 = 高峰前 15 分钟会有大量超时
- 建议：早高峰前 30 分钟自动扩到日峰值的 80%
```

#### 3.3 零中断架构评估

```
逐项检查：
┌───────────────────┬────────────────────────────────┐
│ 检查项             │ 达标标准                        │
├───────────────────┼────────────────────────────────┤
│ 每个环节实例数     │ ≥ 2个 AZ 各 ≥ 2个实例          │
│ 故障切换自动化     │ 无需人工介入，< 30秒自动切换     │
│ 熔断降级策略       │ 核心链路100%覆盖               │
│ 灰度发布能力       │ 支持 5% → 观察 → 全量           │
│ 健康检查          │ interval ≤ 5s, threshold ≤ 2    │
│ 连接池复用        │ idle timeout > 客户端keepalive  │
└───────────────────┴────────────────────────────────┘

"零中断"标准：任意单组件故障影响 < 0.1% 用户
不达标项 = 风险项，按影响面排序输出
```

#### 3.4 网络三层排查

```
"司机端/乘客端异常"时的排查顺序：

Layer 1: CLB
- 后端实例全部健康？→ 有 unhealthy = CLB 层问题
- 连接数接近上限？→ > 80% = 需扩规格

Layer 2: API 网关
- 限流是否触发？→ 触发率 > 1% = 阈值需调高或扩容
- QPS 配额使用率？→ > 80% = 申请扩配额

Layer 3: TGW/安全组
- 安全组是否有新规则？→ 新增规则可能拦截合法流量
- conntrack 表是否满？→ 满 = 丢包

三层都通 + 仍超时 → 应用层问题（进入调度服务内部排查）
```

## 注意事项

- 出行对延迟极敏感，200ms→1.2s 就是体验崩溃
- 春运前至少 T-14 做容量评估
- 早晚高峰期间禁止变更，只做观测
- 预扩容比事后扩容重要（弹性窗口只有 15 分钟）
- "司机端网络异常"先排移动网络（让司机切 4G / 重启路由器）
